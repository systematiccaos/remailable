AppLoad QML contract (from https://github.com/asivery/rm-appload):
- Root QML file MUST have: `signal close` and `function unloading() { ... }`
- AppLoad external app format: external.manifest.json + icon.png + executable
- External manifest requires: name, application (path to executable)

reMarkable Paper Pro SDK:
- Toolchain installer: remarkable-platform-image-4.0.813-ferrari-public-x86_64-toolchain.sh
- Target: arm-remarkable-linux-gnueabihf (ferrari = Paper Pro codename)
- Source environment: /opt/codex/rm ferrari/.../environment-setup-...

CXX-Qt approach (from https://kdab.github.io/cxx-qt/book/):
- Uses cxx-qt crate + cxx-qt-build in build.rs
- Cargo build script generates C++ bridge code
- QML files loaded via QQmlApplicationEngine
- Requires qmake findable on PATH or via QMAKE env var

  <action>
    1. Create `.cargo/config.toml` with cross-compilation target for the reMarkable Paper Pro.

       The reMarkable Paper Pro uses the "ferrari" codename. The official Yocto SDK installs to:
       `/opt/codex/rm-ferrari/4.0.813/` (exact path varies — check SDK installer output)

       The target triple is `arm-remarkable-linux-gnueabihf` (or `armv7-remarkable-linux-gnueabihf` — determine exact triple from the SDK's environment-setup script).

       Since Rust doesn't have a built-in target for reMarkable, use a custom target JSON file:

       Create `.cargo/arm-remarkable-linux-gnueabihf.json`:
       ```json
       {
         "arch": "arm",
         "data-layout": "e-m:e-p:32:32-Fi8-i64:64-v128:64:128-a:0:32-n32-S64",
         "env": "gnu",
         "features": "+v7,+vfp3,-d32,+thumb2,+neon",
         "linker-flavor": "gcc",
         "os": "linux",
         "vendor": "remarkable",
         "target-family": "unix",
         "target-pointer-width": "32",
         "llvm-target": "arm-remarkable-linux-gnueabihf"
       }
       ```
       Verify this against the actual SDK — the environment-setup script in the SDK defines CC and CXX, which reveal the exact target triple and CPU features.

    2. In `.cargo/config.toml`, configure the target:
       ```toml
       [target.arm-remarkable-linux-gnueabihf]
       linker = "arm-remarkable-linux-gnueabihf-g++"
       rustflags = [
         "-C", "link-arg=--sysroot=/opt/codex/rm-ferrari/4.0.813/sysroots/cortexa72-crypto-remarkable-linux-gnueabi",
         "-C", "link-arg=-L/opt/codex/rm-ferrari/4.0.813/sysroots/cortexa72-crypto-remarkable-linux-gnueabi/usr/lib",
       ]

       [env]
       QMAKE = "/opt/codex/rm-ferrari/4.0.813/sysroots/cortexa72-crypto-remarkable-linux-gnueabi/usr/bin/qmake"
       PKG_CONFIG_PATH = "/opt/codex/rm-ferrari/4.0.813/sysroots/cortexa72-crypto-remarkable-linux-gnueabi/usr/lib/pkgconfig"
       PKG_CONFIG_SYSROOT_DIR = "/opt/codex/rm-ferrari/4.0.813/sysroots/cortexa72-crypto-remarkable-linux-gnueabi"
       ```

       IMPORTANT: The exact paths above are PLACEHOLDERS. The actual SDK installation path is determined by running the toolchain installer. Check:
       - The SDK self-extracting script from: `https://storage.googleapis.com/remarkable-codex-toolchain/3.14.3.0/meta-toolchain-remarkable-4.0.813-ferrari-public-x86_64-toolchain.sh`
       - After sourcing the environment-setup script, $CC, $CXX, $PKG_CONFIG_PATH reveal the exact paths

       The .cargo/config.toml should use path references that match what the SDK installer creates. Add comments explaining how to adjust for different SDK installation locations.

    3. Add the Rust ARM target:
       ```
       rustup target add armv7-unknown-linux-gnueabihf
       ```
       Note: For custom targets defined via JSON, `rustup target add` is not needed — Cargo finds the .json file in .cargo/. However, you may need the `armv7-unknown-linux-gnueabihf` standard library. The alternative is to build with `-Z build-std` for the custom target.

    4. Add a README note about SDK setup requirements (not a full README, just a comment in config.toml).

    NOTE: The actual cross-compilation will NOT be verified in this task (that requires the Yocto SDK installed locally or in CI). The CI pipeline (Plan 02) handles the actual cross-build. This task ensures the configuration files are correctly structured.
