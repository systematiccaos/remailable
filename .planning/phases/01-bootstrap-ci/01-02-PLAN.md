    1. Create `packaging/external.manifest.json`:
       ```json
       {
         "name": "remailable",
         "application": "./remailable",
         "workingDirectory": ".",
         "environment": {
           "QT_QPA_PLATFORM": "eglfs",
           "QT_QPA_EGLFS_INTEGRATION": "eglfs_kms"
         },
         "aspectRatio": "original",
         "disablesWindowedMode": true
       }
       ```
       The environment variables configure Qt to use the eglfs platform plugin with KMS integration, which is the standard way to run Qt apps on the reMarkable Paper Pro without X11/Wayland. These may need adjustment based on actual device behavior.

    2. Create `packaging/icon.png` — a simple placeholder icon (128x128 PNG).
       Use ImageMagick or a simple SVG→PNG conversion:

       ```
       convert -size 128x128 xc:'#4a4a4a' -fill white -gravity center -pointsize 72 -annotate 0 '✉' packaging/icon.png
       ```
       If ImageMagick is not available, create a minimal valid PNG using Python or a downloaded placeholder. The icon is a gray square with a mail symbol — it can be refined later.

    3. Create `scripts/build-rcc.sh` — compiles QML files into a Qt resource file (.rcc):
       ```bash
       #!/bin/bash
       set -euo pipefail

       # Build Qt resource file from QML sources
       # Requires: Qt's rcc tool on PATH (from SDK or host Qt installation)

       SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
       PROJECT_ROOT="$(cd "$SCRIPT_DIR/.." && pwd)"
       QML_DIR="$PROJECT_ROOT/qml"
       OUTPUT_DIR="$PROJECT_ROOT/build"

       mkdir -p "$OUTPUT_DIR"

       # Create a .qrc file listing all QML resources
       cat > "$OUTPUT_DIR/resources.qrc" << 'QRC'
       <RCC>
           <qresource prefix="/">
       QRC

       # Add each QML file to the resource
       find "$QML_DIR" -name "*.qml" -type f | while read -r qml_file; do
           relative="${qml_file#$QML_DIR/}"
           echo "        <file>$relative</file>" >> "$OUTPUT_DIR/resources.qrc"
       done

       cat >> "$OUTPUT_DIR/resources.qrc" << 'QRC'
           </qresource>
       </RCC>
       QRC

       # Compile the .qrc into .rcc binary
       rcc -binary "$OUTPUT_DIR/resources.qrc" -o "$OUTPUT_DIR/resources.rcc"

       echo "Built resources.rcc at $OUTPUT_DIR/resources.rcc"
       ```
       Make it executable: `chmod +x scripts/build-rcc.sh`

       NOTE: For the AppLoad external app format, QML files can also be loaded directly from the filesystem (not compiled into .rcc). But compiling into .rcc is the standard Qt approach and works without depending on file paths at runtime. The approach choice should be documented. Since AppLoad's native app format uses resources.rcc but the external format just needs the binary to run, we can load QML from the filesystem in the binary itself (the QmlApplicationEngine load path).

       REVISED APPROACH: Since we're using the AppLoad "external application" format (running a standalone binary), we don't need resources.rcc at all. The binary itself loads QML files from its working directory. The script should instead just copy QML files into the package directory. Simplify:
       ```bash
       #!/bin/bash
       set -euo pipefail
       # Copy QML files to the package directory
       SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
       PROJECT_ROOT="$(cd "$SCRIPT_DIR/.." && pwd)"
       PKG_DIR="$PROJECT_ROOT/build/remailable"
       mkdir -p "$PKG_DIR"
       cp -r "$PROJECT_ROOT/qml" "$PKG_DIR/qml"
       echo "Copied QML files to $PKG_DIR/qml"
       ```

    4. Create `scripts/package.sh` — assembles the final AppLoad package directory:
       ```bash
       #!/bin/bash
       set -euo pipefail

       # Assemble AppLoad-compatible package for reMarkable Paper Pro
       # Output: build/remailable/ directory ready for deployment

       SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
       PROJECT_ROOT="$(cd "$SCRIPT_DIR/.." && pwd)"
       BINARY_PATH="${1:-$PROJECT_ROOT/target/arm-remarkable-linux-gnueabihf/release/remailable}"
       PKG_DIR="$PROJECT_ROOT/build/remailable"

       echo "Assembling AppLoad package..."
       rm -rf "$PKG_DIR"
       mkdir -p "$PKG_DIR"

       # Copy the cross-compiled binary
       cp "$BINARY_PATH" "$PKG_DIR/remailable"
       chmod +x "$PKG_DIR/remailable"

       # Copy AppLoad manifest (external app format)
       cp "$PROJECT_ROOT/packaging/external.manifest.json" "$PKG_DIR/external.manifest.json"

       # Copy icon
       cp "$PROJECT_ROOT/packaging/icon.png" "$PKG_DIR/icon.png"

       # Copy QML files (loaded by binary at runtime from working directory)
       mkdir -p "$PKG_DIR/qml"
       cp "$PROJECT_ROOT/qml/"*.qml "$PKG_DIR/qml/"
       cp "$PROJECT_ROOT/qml/"*.js "$PKG_DIR/qml/" 2>/dev/null || true

       # Create tarball for easy deployment
       cd "$PROJECT_ROOT/build"
       tar czf remailable-appload.tar.gz -C . remailable/

       echo "Package assembled at: $PKG_DIR"
       echo "Tarball: $PROJECT_ROOT/build/remailable-appload.tar.gz"
       echo ""
       echo "To install on device, copy the remailable/ directory to:"
       echo "  /home/root/xovi/exthome/appload/remailable/"
       ```
       Make it executable: `chmod +x scripts/package.sh`

    5. Update the `src/main.rs` QML load path to work with the AppLoad package structure:
       When running as an AppLoad external app, the working directory is set to the app directory. So the QML path should be `qml/main.qml` (relative to the working directory). Verify this works with the QML engine load call from Plan 01.

    6. Update `.gitignore` to include `build/` and `*.tar.gz` if not already present.

  <action>
    Create `.github/workflows/build.yml` with a CI pipeline that:

    1. **Triggers on push to main** and on manual workflow_dispatch (for release builds).

    2. **Runs on ubuntu-latest** (Ubuntu 22.04 or latest — the Yocto SDK is Linux-only).

    3. **Installs Rust toolchain** using `dtolnay/rust-toolchain@stable` with the `armv7-unknown-linux-gnueabihf` target (closest standard target — the custom JSON overrides some settings).

    4. **Downloads and installs the reMarkable Paper Pro Yocto SDK**:
       ```yaml
       - name: Cache Yocto SDK
         id: cache-sdk
         uses: actions/cache@v4
         with:
           path: /opt/codex/rm-ferrari
           key: remarkable-ferrari-sdk-4.0.813

       - name: Install reMarkable Paper Pro SDK
         if: steps.cache-sdk.outputs.cache-hit != 'true'
         run: |
           wget -q https://storage.googleapis.com/remarkable-codex-toolchain/3.14.3.0/meta-toolchain-remarkable-4.0.813-ferrari-public-x86_64-toolchain.sh
           echo "29e2cce4280dbde182937c9614681a59a14aca8a18e06957bf66f8aeb0f3b968  meta-toolchain-remarkable-4.0.813-ferrari-public-x86_64-toolchain.sh" | sha256sum -c
           chmod +x meta-toolchain-remarkable-4.0.813-ferrari-public-x86_64-toolchain.sh
           ./meta-toolchain-remarkable-4.0.813-ferrari-public-x86_64-toolchain.sh -y -d /opt/codex/rm-ferrari
       ```

    5. **Sources the SDK environment and builds**:
       ```yaml
       - name: Source SDK environment
         run: |
           # Find and source the environment setup script
           source /opt/codex/rm-ferrari/*/environment-setup-*

       - name: Build for reMarkable Paper Pro
         run: |
           source /opt/codex/rm-ferrari/*/environment-setup-*
           export QMAKE="$(which qmake)"
           export PKG_CONFIG_PATH="$OECORE_PKGS_PATH"
           export PKG_CONFIG_SYSROOT_DIR="$OECORE_TARGET_SYSROOT"
           cargo build --release --target arm-remarkable-linux-gnueabihf
         env:
           CARGO_TARGET_ARM_REMARKABLE_LINUX_GNUEABIHF_LINKER: arm-remarkable-linux-gnueabihf-g++
       ```

       IMPORTANT: The exact environment variables set by the SDK (`OECORE_PKGS_PATH`, `OECORE_TARGET_SYSROOT`, etc.) are defined in the environment-setup script. After sourcing it, $CC, $CXX, $PKG_CONFIG_PATH, and $OECORE_* variables are available. The CI step must source the script in the same shell step as the build (GitHub Actions `run` blocks are separate shells).

       NOTE: If the custom target JSON is needed, add:
       ```yaml
       - name: Copy custom target specification
         run: |
           mkdir -p ~/.rustup/toolchains/stable-x86_64-unknown-linux-gnu/lib/rustlib/
           cp .cargo/arm-remarkable-linux-gnueabihf.json target_spec.json || true
       ```
       Actually, Cargo will find the target JSON in `.cargo/` automatically when building with `--target arm-remarkable-linux-gnueabihf`.

       NOTE ON CXX-QT BUILD: The `cxx-qt-build` crate runs Qt's moc and other tools at build time. The SDK provides these tools. Ensure QMAKE environment variable points to the SDK's qmake before running cargo build.

    6. **Packages the AppLoad app**:
       ```yaml
       - name: Package for AppLoad
         run: |
           ./scripts/package.sh target/arm-remarkable-linux-gnueabihf/release/remailable
       ```

    7. **Uploads the build artifact**:
       ```yaml
       - name: Upload AppLoad package
         uses: actions/upload-artifact@v4
         with:
           name: remailable-appload
           path: build/remailable-appload.tar.gz
           retention-days: 30
       ```

    8. **Also add a native check step** (build on host to catch Rust compile errors early):
       ```yaml
       - name: Check (native)
         run: cargo check
       ```
       Run this BEFORE the cross-compilation step so quick compile errors are caught early.

    9. **Add Qt host dependencies** — CXX-Qt build requires Qt development headers on the CI runner:
       ```yaml
       - name: Install Qt (host)
         run: |
           sudo apt-get update
           sudo apt-get install -y qt6-base-dev qt6-declarative-dev
       ```

       This is needed because `cargo check` (native) requires Qt headers even on the host.

    IMPORTANT NOTES:
    - The CI pipeline may need iteration — the exact SDK paths and environment variables won't be known until the SDK is actually installed. The workflow should be written with best-guess paths and clear comments about what needs adjustment.
    - Use `set -euo pipefail` in all shell steps
    - The `source` command in GitHub Actions requires using `bash` shell (default is `bash` for `run:`)
    - GitHub Actions `run:` starts a new shell each step — variables from `source` do NOT persist between steps. Use `echo "VAR=value" >> $GITHUB_ENV` to persist env vars between steps, OR keep the build in a single step.
